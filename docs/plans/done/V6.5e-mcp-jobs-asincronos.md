# V6.5e — MCP: trabajos asíncronos (jobs) — las operaciones largas no matan la sesión

**Contexto / evidencia (perezosa 66, 2026-07-16)**: dos lotes MCP (`run_batch` de 12
run_scripts, `edit_batch` de 3) superaron el timeout del cliente → `MCP error -32001` →
el servidor TERMINÓ bien pero el agente quedó ciego («¿aplicó?») y tuvo que adivinar con
`/api/health`. Para no repetirlo abandonó el MCP y se fue a REST crudo con scripts, donde
perdió las protecciones del MCP (PATCH de `edit_command`, contratos, «¿quisiste decir…?»)
y metió el bug del `merge` en body (el default REST es replace) → dos travesaños
destruidos a cajas default. **El modo de fallo actual empuja al agente al camino inseguro
exactamente en los modelos grandes** — contradice el mandato de escala.

Hay DOS capas de timeout y las dos matan igual: `_api()` del cliente fino usa
`httpx.Client(timeout=120)` (mcp_server.py:51) y el host MCP corta la tool a ~180 s.
Subir números no arregla nada (siempre habrá un lote más lento); el fix estructural es
que **ninguna mutación larga viva dentro de una request**.

**Diseño en una línea**: las mutaciones por lote se ENCOLAN como *job* con recibo
(`job_id`); el cliente fino espera con presupuesto y, si no llega a tiempo, devuelve el
recibo en vez de un error — el resultado nunca se pierde, solo se difiere.

## 1. Servidor: `core/apolo/api/jobs.py` (módulo nuevo) + endpoints

**JobStore** (responsabilidad única, testeable aislado, sin FastAPI):
- Cola FIFO con **UN worker thread** (daemon). No hay paralelismo que ganar: STATE_LOCK
  serializa igual; un solo worker da orden determinista (filosofía «un lote = UN
  regenerate»).
- Job: `{id (uuid4 corto), tipo ("run_batch"|"edit_batch"), estado
  (encolado|corriendo|ok|error), resultado (payload dict), error (str), http_status,
  creado, terminado}`.
- `submit(tipo, fn) -> job_id`; el worker ejecuta `fn()` (el MISMO closure que hoy corre
  el endpoint), captura `HTTPException` → `estado=error` + `error=detail` +
  `http_status` (un ContractError se ve IGUAL que el 400 de hoy, solo que diferido).
- `get(job_id, wait_s=0..30)` → long-poll sobre `threading.Condition` (despierta al
  terminar o al vencer wait_s).
- Retención: los últimos 20 TERMINADOS (FIFO eviction); encolado/corriendo jamás se
  desalojan. Sin persistencia: un reload los pierde (§3).
- **Regla de locks**: el lock del store es HOJA — jamás sostenerlo mientras se llama a
  código del documento. El worker llama `_state_or_error(...)` (que toma STATE_LOCK) sin
  ningún lock propio, y solo DESPUÉS escribe el resultado en el store.

**api/main.py**:
- `POST /api/commands/batch` y `PATCH /api/commands/batch` aceptan query
  `?async=true` (`async` es keyword de Python → `Query(alias="async")`). Con él:
  `202 {job_id, estado: "encolado"}` al instante. **Sin él: byte-idéntico a hoy** (los
  clientes existentes ni se enteran). El closure encolado es EXACTAMENTE el `_run` +
  `_state_or_error` actuales — misma atomicidad, mismo undo, mismos contratos, mismo
  autosave, cero código duplicado.
- `GET /api/jobs/{id}?wait_s=25` → `{job_id, tipo, estado, error?, resultado?}`;
  desconocido → 404 con el mensaje honesto de §3. `GET /api/jobs` → brief de los
  últimos 20.
- **VERIFICAR antes de codificar**: cómo notifica `_state_or_error` por WebSocket y cómo
  dispara `_autosave()` — desde el hilo del worker ese camino debe ser thread-safe (si
  toca un loop asyncio → `call_soon_threadsafe`; el `_AutosoveScheduler` ya es Timer-based
  y debería tolerarlo). Si algo no es thread-safe, arreglarlo AHÍ (fuente única), no
  duplicar un camino paralelo para jobs.

## 2. Cliente fino: enviar-y-esperar con recibo AUTOMÁTICO (mcp_server.py)

La clave de calibración: **el camino seguro no puede ser opt-in** — el agente no sabe de
antemano cuánto tardará un lote. `run_batch`/`edit_batch` SIEMPRE envían `?async=true` y
esperan ellos:

- Helper `_submit_and_wait(method, path, body, params, detail)` → envía, recibe
  `job_id`, y hace polls `GET /api/jobs/{id}?wait_s=20` en bucle hasta un presupuesto
  total `APOLO_MCP_WAIT_S` (env, default 90 s — colchón bajo los 120 de httpx y los
  ~180 del host; cada poll individual es corto, el httpx de 120 queda como está).
- Job terminó a tiempo → la tool devuelve **exactamente lo de hoy** (`_scene_brief`
  del resultado + `contrato`); `estado=error` → `RuntimeError` con el detail (UX
  idéntica al 400 actual).
- No terminó → devuelve `{"job": id, "estado": "corriendo", "seguir": "llama
  get_job('<id>')"}` — recibo, no error.
- **Tool nueva `get_job(job_id, detail="diff", wait_s=20)`** (69→70 tools; recordar en
  el cierre que el host MCP debe REINICIARSE): re-consulta el job; resultado cacheado →
  re-preguntar es siempre seguro. Docstring con la semántica completa (recibo, reload →
  404 → verificar con get_scene, nunca reintentar el lote a ciegas).

## 3. Semántica de fallo HONESTA

- **404 de job desconocido** (reload de por medio): el mensaje debe decir «el servidor
  pudo reiniciarse; el lote PUDO haber aplicado (el autosave lo habría guardado) —
  verifícalo con get_scene/health antes de reintentar». Peor caso = el statu quo de hoy;
  nunca peor.
- Contrato fallido → `estado=error`, doc intacto (mismo texto del ContractError).
- Cancelación: FUERA DE ALCANCE (no se puede interrumpir OCCT a media regeneración de
  forma segura). Un job encolado-no-iniciado tampoco se cancela en V6.5e (YAGNI).
- Shutdown/reload con job corriendo: muere con el proceso — igual que hoy muere la
  request larga; la durabilidad del autosave ya cubre el disco.
